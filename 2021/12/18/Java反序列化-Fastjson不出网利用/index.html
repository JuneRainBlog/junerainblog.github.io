    Fastjson不出网利用

    在Fastjson这个反序列化漏洞中是使⽤ TemplatesImpl 和 JdbcRowSetImpl 、 BasicDataSource 构造恶意
    代码实现命令执⾏。

    第⼀个利⽤链是常规的Java字节码的执⾏，但是需要开启Feature.SupportNonPublicField，，否则不支持私有属性的传入，存在局限性；
    第⼆个利⽤链⽤到的是JNDI注⼊，利⽤条件相对较低，但是需要连接远程恶意服务器，在⽬标没外⽹的情况下⽆法直接利
    ⽤；
    第三个利⽤链也是⼀个字节码的利⽤，但其⽆需⽬标额外开启选项，也不⽤连接外部服务器，利⽤条件更低

    0x01 TemplatesImpl利用链

    1.1 利⽤条件

    1、服务端使⽤parseObject()时，必须使⽤如下格式才能触发漏洞：
    JSON.parseObject(input, Object.class, Feature.SupportNonPublicField);
    2、服务端使⽤parse()时，需要 JSON.parse(text1,Feature.SupportNonPublicField);
    这是因为payload需要赋值的⼀些属性为private属性，服务端必须添加特性才会去从json中恢复private属性的数据

    1.2 影响版本

    Fastjson 1.2.22-1.2.24、JDK1.7u21版本

    1.3 漏洞复现

    POC：

    import com.alibaba.fastjson.JSON;
import com.alibaba.fastjson.parser.Feature;

    1._bytecodes为什么需要base64编码？

    位置：DefaultFieldDeserializer.parseField() 71行左右，调用了deserialze函数，跟进

    位置：com/alibaba/fastjson/serializer/ObjectArrayCodec.deserialze() 128行

    位置：com/alibaba/fastjson/parser/JSONScanner.bytesValue() 112行

    进行了base64解码操作

    2._tfactory为什么为{}

    如果我们不传入的话，在最后调用的时候，由于_tfactory 为null，所以直接返回了，就到不了加载字节码的地方，所以我们得想办法传入一个数值让_tfactory最后不为null，而是为TransformerFactoryImpl 实例

    poc中传入的是{} ，让我们来看一下为什么可以

    如果传入的键值为空的话，如果为空会根据类属性定义的类型自动创建实例

    1.5 POC分析

    {
"@type":"com.sun.org.apache.xalan.internal.xsltc.trax.TemplatesImpl",
"_bytecodes":["evilCode"],
'_name':'aaa',
'_tfactory':{ },
"_outputProperties":{ }
}

    @type
    指定的解析类，这⾥即 com.sun.org.apache.xalan.internal.xsltc.trax.TemplatesImpl ，Fastjson根据
    指定类去反序列化得到该类的实例，在默认情况下只会去反序列化public修饰的属性，在poc中， bytecodes
    与 _name 都是私有属性，所以要想反序列化这两个，需要在 parseObject() 时设
    置 Feature.SupportNonPublicField
    _bytecodes
    是我们把恶意类的.class⽂件⼆进制格式进⾏base64编码后得到的字符串
    _outputProperties
    漏洞利⽤链的关键会调⽤其参数的getOutputProperties⽅法导致命令执⾏
    _tfactory:{}
    在defineTransletClasses()时会调⽤getExternalExtensionsMap(),当为null时会报错，所以要对tfactory
    设值

    0x02 BCEL利⽤链—org.apache.tomcat.dbcp.dbcp2.BasicDataSource

    2.1 利⽤条件

    • ⽆需⽬标额外开启选项，也不⽤连接外部服务器
    • 但需要应⽤部署在Tomcat应⽤环境中，因为Tomcat应⽤环境⾃带tomcat-dbcp.jar
    • 对于SpringBoot这种⾃带Tomcat可以直接以单个jar⽂件部署的需要在maven中配置tomcat-dbcp。
    • ⽽且对于不同的Tomcat版本使⽤的poc也不同
    • Tomcat 8.0以后使⽤ org.apache.tomcat.dbcp.dbcp2.BasicDataSource
    • Tomcat 8.0以前使⽤ org.apache.tomcat.dbcp.dbcp.BasicDataSource
    • JDK版本< 8u251，在Java 8u251以后，bcel类被删除

    2.2 影响版本

    Fastjson 1.2.22-1.2.24

    利⽤环境
    jdk1.8.0_102、fastjson1.2.24、tomcat-dbcp9.0.20

    @type

    指定的解析类，这⾥即 org.apache.tomcat.dbcp.dbcp2.BasicDataSource ，Fastjson根据指定类去反序列化得到该类的实例。

    driverClassLoader

    指定Classloade为com.sun.org.apache.bcel.internal.util.ClassLoader

    driverClassName
    设置ClassName为 BCEL… 这种格式后，在 newInstance ⽅法执⾏后被实例化。第⼆个参数 initial 为true时，类加载后将会直接执⾏ static{} 块中的代码

    2.4 调试分析

    这里开发者是用了代码中是调用的是 JSON.parse() 。

    原PoC中很巧妙的利用了 JSONObject对象的 toString() 方法实现了突破。JSONObject是Map的子类，在执行toString() 时会将当前类转为字符串形式，会提取类中所有的Field，自然会执行相应的 getter 、is等方法。

    首先，在 {“@type”: “org.apache.tomcat.dbcp.dbcp2.BasicDataSource”……} 这一整段外面再套一层{}，反序列化生成一个 JSONObject 对象。

    然后，将这个 JSONObject 放在 JSON Key 的位置上，在 JSON 反序列化的时候，FastJson 会对 JSON Key 自动调用 toString() 方法：

    BasicDataSource的toString()方法会遍历这个类的所有getter并执行，于是调用了getConnection方法

    继续跟进 createDataSource函数，他主要调⽤ createConnectionFactory

    跟进可以看到在 createConnectionFactory 函数使⽤ Class.forName 加载类

    在 Class.forName 时，会触发 static 代码块中的代码，从⽽达到任意代码执⾏的⽬的。

    当然，如果目标环境的开发者代码中是调用的是 JSON.parseObject() ，那就不用这么麻烦了。与 parse() 相比，parseObject() 会额外的将 Java 对象转为 JSONObject 对象，即调用 JSON.toJSON()，在处理过程中会调用所有的 setter 和 getter 方法。

    所以对于 JSON.parseObject()，直接传入这样的Payload也能触发：
